# ASR 跨端功能一致性验证标准

**文档版本**: v1.0
**创建日期**: 2026-03-29
**适用范围**: Android + iOS ASR 功能验证

## 验证目标

确保 Wishpool 三端应用在 ASR（自动语音识别）功能上提供一致的用户体验，包括识别准确性、交互流程、错误处理等方面。

## 技术基线对比

### 共同技术栈
| 组件 | Android | iOS | 一致性要求 |
|------|---------|-----|-----------|
| ASR 引擎 | Sherpa ONNX | Sherpa ONNX | ✅ 完全一致 |
| 模型 | zipformer-zh-14M (24.5MB) | zipformer-zh-14M (24.5MB) | ✅ 完全一致 |
| 采样率 | 16kHz | 16kHz | ✅ 完全一致 |
| 处理方式 | 端侧实时 | 端侧实时 | ✅ 完全一致 |
| **交互模式** | **双模式（v0.3.4+）** | **双模式** | ✅ 完全一致 |

### 双模式交互对比（Android v0.3.4+ 重要更新）
| 交互方式 | Android | iOS | 验证重点 |
|---------|---------|-----|---------|
| 单击FAB | DirectPublishSheet | CreateWishDirectSheet | 语音直发流程一致 |
| 长按FAB | PublisherSheet | CreateWishSheetWithASR | 转录编辑流程一致 |
| 直发状态文案 | fromDirect() | directModeStatusText | "完成后直接发送"等提示一致 |
| 编辑状态文案 | from() | 标准状态文案 | "已听到你的心愿✨"等提示一致 |
| 自动提交逻辑 | AsrState.Result → onSubmit | ASRState.result → onSubmit | 直发模式无编辑步骤 |

### 状态映射对比
| 状态 | Android AsrState | iOS ASRState | 验证重点 |
|------|-----------------|-------------|---------|
| 空闲 | `Idle` | `.idle` | 初始状态一致 |
| 权限 | `PermissionRequired` | `.permissionRequired` | 权限处理流程 |
| 下载 | `Downloading(progress)` | `.downloading(progress)` | 进度同步显示 |
| 录音 | `Recording(partialText)` | `.recording(partialText)` | 实时转录一致性 |
| 处理 | `Processing(partialText)` | `.processing(partialText)` | 处理状态过渡 |
| 结果 | `Result(text)` | `.result(text)` | 最终结果一致性 |
| 错误 | `Error(message)` | `.error(message)` | 错误处理一致性 |

## 验证矩阵

### 1. 功能对等性验证

#### 1.1 基础功能对等
- [ ] **权限申请**: 两端都能正确申请和处理麦克风权限
- [ ] **模型管理**: 模型下载、缓存、加载逻辑一致
- [ ] **录音控制**: 开始/停止录音的触发和响应
- [ ] **实时转录**: 录音过程中的文字实时显示
- [ ] **结果处理**: 最终识别结果的获取和展示

#### 1.2 双模式交互对等（Android v0.3.4+ 核心特性）
- [ ] **FAB单击行为**: 单击触发语音直发模式，体验一致
- [ ] **FAB长按行为**: 长按触发语音转录编辑模式，体验一致
- [ ] **直发模式流程**: 录音 → 自动提交，无编辑步骤
- [ ] **编辑模式流程**: 录音 → 转录显示 → 可编辑 → 手动提交
- [ ] **状态文案一致**: 直发模式"完成后直接发送"，编辑模式"已听到你的心愿✨"
- [ ] **录音状态指示**: 录音中的视觉反馈（红点动画等）
- [ ] **进度显示**: 模型下载、语音处理进度的显示方式
- [ ] **错误提示**: 各种错误场景的用户提示

### 2. 识别准确性验证

#### 2.1 标准测试语音
使用以下标准语音进行跨端对比测试：

**短语测试集**:
- "我想去海边放松一下"
- "希望能买一台新电脑"
- "计划明年春天去日本旅游"
- "想学习做烘焙甜品"
- "希望能够身体健康工作顺利"

**长句测试集**:
- "我一直想去欧洲旅行，特别是法国和意大利，希望能在明年夏天的时候实现这个愿望，预算大概在三万元左右"
- "想要学习一门新的编程语言，比如Python或者Swift，然后用它来开发一个小项目，时间安排在接下来的六个月内"

#### 2.2 准确性评判标准
- **完全匹配**: 识别结果与标准文本100%一致 → 优秀
- **语义一致**: 个别字词不同但语义相同 → 良好
- **语义相近**: 主要意思表达正确，细节有偏差 → 可接受
- **语义偏离**: 主要意思理解错误 → 不可接受

#### 2.3 跨端一致性要求
- Android 和 iOS 对同一语音的识别结果**语义一致率 ≥ 90%**
- 字符级相似度 ≥ 85%（使用 Levenshtein 距离计算）

### 3. 性能一致性验证

#### 3.1 响应时间对比
| 操作 | Android 目标 | iOS 目标 | 验证方法 |
|------|-------------|---------|---------|
| 启动录音 | < 1秒 | < 1秒 | 从点击到开始录音的延迟 |
| 实时转录 | < 500ms | < 500ms | 语音到文字显示的延迟 |
| 完成识别 | < 2秒 | < 2秒 | 停止录音到最终结果的时间 |
| 模型加载 | < 3秒 | < 3秒 | 首次使用时模型初始化时间 |

#### 3.2 资源占用对比
- **内存占用**: Android/iOS 模型运行时内存增量应控制在 ±20% 范围内
- **CPU 占用**: 录音识别过程中 CPU 使用率相近
- **电量消耗**: 相同时长语音识别的电量消耗差异 < 30%

- [ ] 引导用户去设置页面的流程一致
- [ ] 权限恢复后的自动重试逻辑一致

#### 4.2 网络异常场景
- [ ] 模型下载失败时的重试机制一致
- [ ] 离线模式的降级策略一致
- [ ] 网络恢复后的自动恢复逻辑一致

#### 4.3 设备兼容性场景
- [ ] 低端设备上的性能降级策略一致
- [ ] 存储空间不足时的处理方式一致
- [ ] 不支持的设备上的友好提示一致

## 验证流程

### 验证环境准备
1. **Android**: API 28+ 测试设备，包含高中低端机型
2. **iOS**: iOS 13.0+ 测试设备，包含不同代际的 iPhone/iPad
3. **测试工具**: 录音设备、性能监控工具、自动化测试脚本

### 验证步骤

#### Step 1: 功能完整性验证
1. 在两个平台上完成 ASR 功能的完整实现
2. 验证所有状态转换和 UI 交互正常工作
3. 确保基础功能无阻塞性 Bug

#### Step 2: 准确性基准验证
1. 使用标准测试语音集进行识别测试
2. 记录每个平台的识别结果
3. 计算跨端一致性指标，确保达到要求

#### Step 3: 性能基准验证
1. 测量各项性能指标（启动时间、响应时间、资源占用）
2. 对比两端性能数据，确保差异在可接受范围内
3. 在不同设备等级上验证性能表现

#### Step 4: 用户体验验证
1. 进行真实用户场景的端到端测试
2. 验证完整的语音输入到愿望提交流程
3. 确保两端用户体验的一致性

#### Step 5: 边界场景验证
1. 测试各种异常和边界场景
2. 验证错误处理和恢复机制
3. 确保两端在异常情况下的行为一致

## 验证通过标准

### 🟢 通过标准
- [ ] 功能对等性：所有核心功能在两端都正常工作
- [ ] 识别准确性：语义一致率 ≥ 90%，字符相似度 ≥ 85%
- [ ] 性能表现：关键指标差异 ≤ 20%
- [ ] 错误处理：异常场景下行为一致，用户提示统一
- [ ] 用户体验：端到端流程顺畅，交互逻辑一致

### 🟡 部分通过（需优化）
- 功能基本可用，但在某些细节上存在差异
- 识别准确性或性能指标略低于标准，但不影响基本使用
- 需要在下个版本中进行优化完善

### 🔴 不通过（需修复）
- 核心功能存在明显缺陷或平台间重大差异
- 识别准确性或性能严重不达标
- 用户体验存在阻塞性问题

## 持续改进机制

### 定期验证
- **每月验证**: 功能回归测试，确保现有功能稳定
- **版本验证**: 每个版本发布前的完整验证
- **新设备验证**: 新设备型号发布时的兼容性验证

### 数据驱动优化
- 收集用户实际使用中的语音识别数据
- 分析识别错误的模式，针对性优化
- 基于用户反馈持续改进用户体验

### 技术栈演进
- 跟踪 Sherpa ONNX 的版本更新和性能优化
- 评估新模型对识别准确性的提升
- 保持两端技术栈的同步演进
