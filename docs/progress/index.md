## 当前产品能力层结论（已对齐）

### 当前阶段产品定位

Wishpool 当前阶段不应再被理解为：
- 活动推荐产品
- 泛社交产品
- 万能许愿平台

更准确的当前定位是：

**面向城市白领的周末解决方案系统**

特点：
- 聚焦中复杂度
- 个性化
- 解决方案化
- 先以“单人也能成行”为主路径
- 在部分场景中，“人”是体验的一部分

### 当前 MVP 成功标准

**真实线下体验完成，才算成功。**

不以以下作为最终成功：
- 浏览内容
- 表达兴趣
- 获得方案
- 初步撮合成功
- 完成支付

### 当前最优先验证的能力

优先级最高的是：
- `推进与校验`
- `协同锁定`

不是优先验证：
- 首页消费时长
- 泛 feed 内容丰富度
- 纯 AI 方案美观程度

### AI / 人工分工结论

当前方向不是纯 AI 自动执行，也不是纯人工 concierge，而是：

**AI 驱动、人工把关的推进系统**

- `AI`：负责高频、标准化的推进动作
- `人工`：负责专家知识 check、线下现实验证、异常判断、关键节点兜底
- `用户`：负责表达意图、补充约束、确认关键决策、最终参与体验

## 通用产品主链路（后续重构的业务骨架）

本次重构后，前端工程应围绕以下主链路组织，而不是围绕 Demo 页面顺序组织：

1. `意图进入`
2. `需求转译`
3. `候选方案生成`
4. `可行性校验`
5. `协同锁定`
6. `成行推进`
7. `体验完成`
8. `结果沉淀`

### 关键判断

- 阶段 2 的产出应该是“任务单”，不是聊天记录
- 阶段 4 是产品差异化核心之一，因为体现 AI + 人工协同
- 阶段 7 才是当前 MVP 的成功终点
- 用户应“轻确认”候选方向，再在校验后做正式确认

## 当前代码问题（本次重构的直接原因）

### 1. 工程基线不可信

已确认问题：
- `demo/package.json` 中 `pnpm run check` 当前失败
- 失败点在：`demo/client/src/const.ts`
- 当前代码仍引用不存在的 `@shared/const`

### 2. 配置与目录结构不一致

当前配置仍保留旧脚手架/残留假设：
- `demo/tsconfig.json`
- `demo/vite.config.ts`
- `demo/package.json`

存在的问题包括：
- 假定存在 `shared/**/*`
- 假定存在 `server/**/*`
- 假定存在 `server/index.ts`
- alias 与当前仓库真实目录不完全匹配

### 3. 主实现仍是单文件 Demo 结构

当前核心实现主要集中在：
- `demo/client/src/pages/WishpoolDemo.tsx`

问题不是“文件大”本身，而是：
- 页面叙事
- 主题切换
- 屏幕顺序
- 业务状态
- 动效与内容数据

仍然纠缠在一起，不利于后续承接真实功能。

## 本次重构的结构性目标

### 目标 1：恢复工程可信度

先让项目具备最基本的工程可信度：
- `check` 可运行
- 配置与实际目录一致
- 不再依赖当前不存在的共享模块或服务端目录

### 目标 2：按产品能力拆结构

未来前端结构不再按“Demo 页面”拆，而应逐步转向：
- `意图 / 任务`
- `方案 / 校验`
- `推进 / 锁定`
- `履约 / 结果`

### 目标 3：表达层与能力层分离

- `V1 / V2 / V2.1` 的页面表达层继续存在
- 但产品主逻辑不再直接绑定在页面切换顺序上

### 目标 4：为真实开发预留底座

重构后应能更自然承接：
- 真实接口
- 真实任务状态
- 人工校验节点
- 推进记录
- 协同锁定信息

## 建议中的未来前端结构（方向性，不是最终代码清单）

后续前端应逐步往以下结构收敛：

- `app`
  - 应用入口、Provider、全局壳层
- `domains`
  - 业务核心对象与状态模型
- `features`
  - 面向动作的组合能力
- `shared`
  - 通用 UI、hooks、工具、常量
- `demo-flow`（或同类表达层）
  - 承接现有 Demo narrative 和页面组织

### 建议中的核心业务对象

- `WishTask`
- `CandidatePlan`
- `ValidationCheck`
- `ExecutionStep`
- `LockDecision`
- `ExperienceRecord`

### 建议中的任务状态模型

- `draft`
- `clarifying`
- `planning`
- `validating`
- `locking`
- `ready`
- `in_progress`
- `completed`
- `failed`
- `cancelled`

关键原则：
- 页面是状态的表现层
- 状态不是页面枚举

## 本次重构的分阶段执行方案

### Phase 0：工程基线修复

目标：
- 先让项目从“脚手架残留状态”恢复到“可被信任的前端工程状态”

重点：
- 修复 `const.ts` 中无效共享依赖
- 对齐 `tsconfig / vite.config / package.json`
- 让当前目录结构与配置一致

### Phase 1：拆开 Demo 页面流与业务状态

目标：
- 不再让 `WishpoolDemo.tsx` 独占全部产品逻辑

重点：
- 将 narrative 页面流与业务状态流转解耦
- 保留现有效果，但减少“页面顺序即产品逻辑”的耦合

### Phase 2：建立任务闭环骨架

目标：
- 落地最小产品能力模型

重点：
- 引入任务对象、方案对象、校验对象、锁定对象
- 为未来真实状态流转打底

### Phase 3：建立推进与协同视角

目标：
- 为后续最重要的 3/4 能力服务

重点：
- 用户视角：查看进展、补确认
- 平台视角：校验、推进、锁定
- AI 视角：摘要、建议动作、状态归纳

### Phase 4：样本场景验证

目标：
- 用三个例子验证通用模型是否成立

重点：
- 不再为每个场景单独长出一套结构
- 用同一套能力模型承接三种不同复杂度/情绪类型
