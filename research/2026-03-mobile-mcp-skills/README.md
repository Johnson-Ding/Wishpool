# 原生 iOS / Android MCP 与 Skills 调研归档

> 日期：2026-03-28
> 调研范围：原生 iOS / Android 工具链
> 当前状态：初筛完成，待继续复核

## 1. 归档目标

本归档用于沉淀一轮随时发起、随时继续的移动端工具调研，目标不是写成技术白皮书，而是回答下面几个实际问题：

1. 当前有哪些 MCP 和 Skills 适合原生 iOS / Android 开发
2. 哪些工具更适合作为开发主链路，哪些更适合作为测试或辅助能力
3. 哪些候选项值得继续深入验证，哪些应暂时排除

本归档不展开接口字段，也不把社区讨论直接当结论。核心是给后续决策留下一套可继续复用的判断基础。

## 2. 本轮范围

### 2.1 纳入范围

- 原生 iOS 开发、构建、调试、模拟器控制
- 原生 Android 开发、构建、调试、模拟器控制
- 移动端自动化测试相关 MCP
- 能直接提升移动端开发效率的 Skills / agent workflows

### 2.2 不纳入范围

- Flutter
- React Native
- Kotlin Multiplatform
- 纯转载文章
- 没有明确官方来源或仓库主体不清晰的项目

## 3. 当前初筛结论

### 3.1 iOS 主线

1. `Apple Xcode 26.3 + 官方 agentic coding / MCP` 值得作为 iOS 原生开发的官方基座继续跟进
2. `XcodeBuildMCP` 是当前最值得重点复核的 iOS 外挂型 MCP，覆盖构建、运行、调试、模拟器与 UI 自动化

### 3.2 Android 主线

1. `Android Studio AI agent + MCP support + Journeys` 值得作为 Android 官方路线继续跟进
2. `gradle-mcp-server` 值得作为 Android 构建与任务执行的补强件继续复核

### 3.3 双端补充

1. `mobile-next/mobile-mcp` 更偏移动端交互与设备自动化能力，适合作为补充，不宜直接当主开发基座
2. `appium/appium-mcp` 更偏自动化测试与回归验证，适合作为测试层补充

### 3.4 当前不建议进入主推荐

1. 只做轻量 `ADB` 命令封装、缺少工程上下文能力的 MCP
2. 已出现明确安全风险通告的候选项

## 4. 当前判断

1. iOS 方向已经出现相对清晰的“官方基座 + 外挂增强”组合
2. Android 方向更像“官方 IDE Agent 能力 + 第三方构建 / 设备能力补强”的组合
3. 跨端类 MCP 更适合放在测试与自动化层，而不是直接承担原生开发主链路

## 5. 建议的后续动作

1. 对 `XcodeBuildMCP` 做更细的能力复核，确认其与 Xcode 26.3 官方能力的边界关系
2. 对 `Android Studio AI agent` 与 `gradle-mcp-server` 做配合方式梳理，判断是否能形成 Android 主推荐组合
3. 对 `mobile-mcp` 与 `appium-mcp` 做角色定位，明确其更适合作为验证层还是开发层补充
4. 单独补一轮安全复核，避免高风险仓库混入推荐清单
